2026-04-13 (MONDAY): LORIKAZZ ANDROID-IOT BOTNET

AUTHOR(S):

- Asher Davila Loranca, Doel Santos, Chris Navarrete 

NOTES:

- We identified and analyzed a botnet operation named Lorikazz that targets Android TV set-top boxes for residential proxy monetization (proxyware). The name Lorikazz comes from a comment label in Jackskid botnet stager scripts, which uninstall the Lorikazz package (com.oreo.mcflurry) as a competitor. This is the first detailed analysis of its dropper infrastructure, APK internals, and C2 architecture.
- Lorikazz shares significant technical overlap with the Kimwolf botnet (also known as AISURU): the bundled ELF binaries contain ENS (Ethereum Name Service) resolution logic ("Empty ENS record"), Tor .onion backup C2, SOCKS5 proxy capabilities, and an HTTP/2-capable networking stack (nghttp2, curl with OpenSSL). This strongly suggests Lorikazz is either operated by the same actor behind Kimwolf, or a different threat actor used the Kimwolf codebase and repurposed it for residential proxy operations instead of DDoS.

DETAILS:

- Lorikazz targets the same device population as Jackskid and Kimwolf: Android TV set-top boxes with exposed ADB (Android Debug Bridge) interfaces.
- Dropper shell scripts delivered via ADB follow a consistent pattern: download an APK via netcat (with toybox/busybox fallback chains), verify the payload's SHA-256 hash before installation, install with multiple pm/cmd fallback methods, disable APK verification, whitelist the package from battery optimization via dumpsys deviceidle, and download a secondary ELF binary via a separate port.
- The operator rotates package names across variants while maintaining the same signing certificate (thumbprint 2a1d96f1b066877812587ac94f45f82dfff5f5f9, created 2026-02-06, issuer C=CN CN=a). Confirmed package aliases include com.oreo.mcflurry, com.android.log_handler_v2, com.android.logcatd, and com.android.boothandler. All scripts actively uninstall com.android.boothandlerv2.
- Two distinct APK architectures exist under the same operation:
-- DexService variant (com.oreo.mcflurry): The C2 communication logic is implemented in Java. DexService contacts a hardcoded Tor .onion C2 (mjsneyhl7gaik3dckdy3ss2vyfr6jfh4eteheu2i4bznt7sbh5zokiyd.onion/check_update) over a local SOCKS5 proxy at 127[.]0.0[.]1:9050, sending device fingerprinting data (CPU_ABI, SDK_INT, DEVICE, MODEL). The C2 returns a configuration with download entries (architecture, URL, hash triplets), which DexService downloads, hash-verifies, and executes. TorService manages the Tor SOCKS proxy by running a bundled Tor binary (libtor.so) with --SocksPort 127[.]0.0[.]1:9050. This variant contains only the Tor binary and no additional native payloads. A ConfigParser class parses a text-based config format using key=value pairs (a=architecture, u=url, h=hash).
-- SDKService variant (com.android.log_handler_v2, com.android.logcatd): The C2 logic is delegated entirely to a bundled native ELF binary disguised as libandroid_runtime.so, executed with the argument "service". SDKService monitors the running process list for "logcat" and restarts the ELF if it is not running, providing a watchdog for native persistence. TorService runs a Tor binary disguised as libdalvik_runtime.so (with --SocksPort 127[.]0.0[.]1:9050). The com.android.log_handler_v2 variant adds optimized Tor flags for IoT devices: --HardwareAccel 1 --AvoidDiskWrites 1 --NumEntryGuards 1 --NumPrimaryGuards 1 --ConnLimit 64 --MaxClientCircuitsPending 16 --DirCache 0.
- The Tor binary (libdalvik_runtime.so / libtor.so) is identical across all three APK variants (SHA-256: 024a427fc94e9bd34d96a447631dff75ae43a1421e174f6081aa5ad97f24fdde), confirming they share the same Tor build.
- The native libandroid_runtime.so binaries (the C2 payload in SDKService variants) contain:
-- A second Tor .onion address: rwbxbmflwm7andgmxeo3my7mqqs6najhou7o6f7xnxjsiuirzcnab4yd.onion (different from the .onion in the DexService variant)
-- ENS resolution capability: the string "Empty ENS record" is present, similar to the Kimwolf botnet's documented use of Ethereum Name Service for decentralized C2 resolution
-- SOCKS5 proxy functionality (socks5h protocol, SOCKS4/5 connect messages)
-- HTTP/2 networking via nghttp2 library, with curl and OpenSSL
-- The bc877d variant's ELF is UPX-packed, while the f07821 variant is uncompressed and renames its process at runtime
- The anti-competition uninstall chain in dropper scripts targets residential proxy SDK packages (com.abcproxy.proxysdk, com.abcproxy.lolsdk, com.n2.systemservice*, com.k.sdk, com.st.stub.app) alongside com.android.docks (Snow botnet), indicating Lorikazz's primary monetization is residential proxy bandwidth rather than DDoS-for-hire.
- Delivery infrastructure uses two IPs: 46[.]33.11[.]154 and 195[.]154.103[.]239. Download ports include 25565 and 25569 for APK delivery, 6969 for ELF binary delivery, and 1337 for the first variant.
- SHA-256 hash verification of downloaded APK payloads before installation is a supply-chain integrity measure not observed in Jackskid or Kimwolf stager scripts.
- A piracy IPTV application named TigerTV (com.android.tigertv), containing Latin American streaming content references and connected to the same dropper IP 46[.]33.11[.]154, suggests the operator may also use piracy TV apps as a social engineering infection vector, similar to the BigPanzi botnet's documented approach. However, no BigPanzi components (pandoraspear, pcdn) were found within the TigerTV APK.

ADDITIONAL DETAILS:

- Lorikazz operates within a highly competitive Android TV/STB botnet ecosystem alongside Jackskid, Kimwolf (AISURU), Snow, and BigPanzi. All of these operations target ADB-exposed devices and actively uninstall each other's packages.
- The Lorikazz name originates from a comment (# lorikazz) in Jackskid's ADB stager scripts, which remove the com.oreo.mcflurry package as a competitor. The scripts analyzed in this investigation do the inverse: they install com.oreo.mcflurry.
- Lorikazz is distinct from Jackskid: Lorikazz uses Tor-based C2 and ENS while Jackskid uses DNS-over-HTTPS and ENS; Lorikazz's primary purpose is residential proxy while Jackskid is DDoS-focused; Lorikazz uses different infrastructure providers (Packet Star / Scaleway vs. DigitalOcean); and the two operations actively remove each other's packages.
- Lorikazz is distinct from BigPanzi despite both targeting Android STBs via piracy apps. BigPanzi uses custom encryption (Blowfish ECB) for C2, deploys dual implants (pandoraspear + pcdn), performs DNS hijacking via hosts file modification, and builds P2P CDN networks. None of these characteristics are present in Lorikazz.
- The earliest observed Lorikazz dropper script (fa72d7d9) was first submitted to VirusTotal on January 24, 2026. The APK signing certificate was created on February 6, 2026. The Tor binary (libdalvik_runtime.so / libtor.so) was first submitted to VT on February 7, 2026.
At the time of analysis, the dropper IP 46[.]33.11[.]154 had 0/94 detections on VirusTotal, and the Lorikazz APK 261305cc (com.oreo.mcflurry) had only 1/77 detections. The com.android.log_handler_v2 variant (bc877d) had 25/77 detections, and the com.android.logcatd variant (f07821) had 30/77 detections.
- Organizations should monitor Tor circuit establishment (port 9050 SOCKS proxy) and ENS resolution traffic from Android TV/STB devices, which are strong behavioral indicators of infection.
