2026-05-28 (THURSDAY): TuxBot v3 Evolution (Akiru) Framework Analysis

AUTHOR(S):

- Chris Navarrete and Asher Davila Loranca

NOTES:

- We identified a previously undocumented modular IoT botnet framework named TuxBot v3 Evolution, self-identified on infected devices as "Akiru."
- The framework was recovered from internal telemetry and includes the full source code (C bot agent, Go CNC server, build scripts, Docker test infrastructure, and exploit VM), compiled binaries for 17 architectures, and automated DDoS performance testing reports.
- The bot binary exploits 16 IoT device families, brute-forces telnet with 1,496 credential pairs, and communicates with the CNC over an encrypted TCP channel.
- The CNC server is written in Go, provides an SSH admin shell, and operates a DDoS-for-hire panel with per-user quotas. 
- OSINT pivoting from the dropper IP 185.10.68[.]127 (FlokiNET, Iceland) links TuxBot to the Keksec/AISURU ecosystem through shared infrastructure with Kaitori v3.9 samples and an AISURU yamux-based Go tool.
- The CNC at 209[.]182.237[.]133 (Singapore) is linked to the dropper via a shared jetross[.]com TLS certificate.
- The framework we found is approximately 70% functional.
- The core infection loop works (scanning, exploitation, persistence, primary C2, DDoS). 
- Broken: exploit VM (magic mismatch), IRC/HTTP fallback C2 (XOR key bug), L7 HTTP methods (never integrated), polymorphic engine (never called), CF/CAPTCHA bypass (dead code, leak test domains cfcybernews[.]eu and captcha.kanfetka[.]site), mining (placeholder).


DETAILS:

Exploitation and Scanning:
- 16 hardcoded exploits target 13 CVEs.
- Two dedicated exploit scanners look for a  RCE-based vulnerability and ADB (port 5555).
- All exploits are downloaded from  http://<C2>/bins/bot.<arch>.
- An embedded exploit VM exists but is broken due to a magic number mismatch (0x4558504C vs 0x54555845), leaving 16 additional exploits unusable. 
- Telnet brute-forcing uses 1,496 credential pairs imported from DDOS-ROOTSEC (XOR key 0xB4). Successful logins are reported to 154[.]6.197[.]43.


DDoS:
- 78 attack vector IDs registered, mapping to 6 optimized handlers: UDP generic (25 vectors), TCP SYN (47 vectors, including all L7 HTTP methods which silently become SYN floods), TCP ACK (2), TCP stomp (1), UDP DNS (2), plus a mining placeholder. 
- The IRC channel dispatches 12 named methods (udp, vse, syn, ack, stomp, greip, greeth, udpplain, bypass, std, socket, dns). 
- 92 method implementations are compiled in but never called.

C2 Protocol:
- Primary channel: encrypted TCP. Handshake sends 0xDEADBE01 + 32-byte X25519 public key. CNC responds with its public key.

Fallback C2 channels:
- DNS TXT queries to c2.tuxbot.local (functional), SHA-512 DGA with seed "<YYYY-MM-DD>-TuxBotv3-Evolution- Seed-2025 <index>" producing 20 domains/day (functional), P2P gossip on port 13337 with Ed25519 signed commands (functional), IRC on port 6667 (broken), HTTP polling (broken).
- The IRC and HTTP channels fail because their table entries were encrypted with the wrong XOR key (0x54 instead of runtime 0xB4).

CNC Server:
- Go binary with three listeners: bot protocol on port 1999/31337 (multiplexed with admin via magic bytes), SSH admin shell on port 2222 (banner: SSH-2.0-CNC-Control-Server, DDoS-for-hire panel with per-user quotas), and Machine API on port 9999 (TCP JSON).

Persistence and Stealth:
- 7 mechanisms: systemd service (sd-pam.service), cron (@reboot + */5), shell profile injection (.bashrc/.profile/.zshrc), hidden backup copies at 3 paths, guardian process with crash backoff, watchdog keepalive, and periodic binary relocation across 21 directories.
- Process name mimicry using 20 system daemon names. Competitor killing via /proc memory scanning for Mirai/QBOT/Vamp/Anime/dvrHelper signatures.
-  Anti-VM scoring (threshold 30), ptrace anti-debug, log wiping, OOM protection (-1000), core dump disabled.

XOR Key Bug:
- 58 table entries use XOR encryption (key 0xDEDEFB4F, effective 0xB4). 9 entries were encrypted with a mismatched key (0xDEDEFBAF, effective 0x54) from the offline tool, causing IRC C2, HTTP C2, and 4 exploit payloads to produce garbage at runtime. These features have never worked.

AISURU/Keksec Relationship:
- Dropper IP 185[.]10.68[.]127 is shared with Kaitori v3.9/Keksec.
- A VT sample (yamux Go binary) communicates with both AISURU IP 194[.]46.59[.]169 and the same dropper.
- The jetross.com TLS cert links the CNC (Singapore) to the dropper (Iceland).
- Developer domain digikalas[.]online resolves to Iranian CDN.

ADDITIONAL DETAILS:

- The framework is statically linked (glibc + libsodium), compiled with GCC 14.2.0 (production) and GCC 11.4.0 (VT debug build), targeting 17 architectures. The source tree includes the Go CNC, MariaDB backend, Docker test networks, QEMU environments, and 254 automated DDoS benchmark reports from January 4-6, 2026.
-Source code and documentation analysis shows the developer used an LLM to port exploit scripts, write C bot modules, and generate CNC Go code. 
- Raw LLM chain-of-thought reasoning was left verbatim in Source files, including self-referential phrases like "If the user insists on all exploits.” 
- The LLM used to develop it also hallucinated multiple times throughout the entire codebase.
-Additionally, every C file carries an identical "educational and authorized security research only" disclaimer consistent with typical LLM safety training.
