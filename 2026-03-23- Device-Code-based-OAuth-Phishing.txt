NOTES:

Active device code phishing campaign impersonating a popular cloud-based file storage service and two prominent electronic signature and document workflow platforms. Instead of harvesting credentials, it abuses Microsoft's legitimate Device Code OAuth flow: victims are shown a verification code and directed to paste it into Microsoft's real login page. The attacker's server captures the resulting OAuth tokens in the background. OAuth tokens grant direct application-level access to the victim's account, giving the attacker long-term access to email, files, and identity. Evasion techniques include obfuscated web payload, DevTools detection, debugger traps and other advanced methods. 

DETAILS:

Microsoft Device Code Phishing — Variant Workflow
- Malicious Link Delivery 
-- Phishing links tailored to specific customers embedded in phishing emails. 
-- Victim clicks it
- Obfuscated HTML Delivery    
-- Attacker server serves a page containing an encrypted payload with an embedded decryption key 
-- The visible shell shows only the legitimate branding
-Client-Side Decryption    
-- Browser uses the embedded key to decrypt the payload in-browser
-- Real page content replaces the placeholder shell
- Device Code Request    
-- Browser sends attacker server a "start session" request
-- Attacker server initiates a Microsoft Device Code flow with Microsoft OAuth
-- Microsoft returns the user code, which the attacker's server then relays to the browser
- Code Displayed to Victim
-- Browser displays user code and instructs victim to: 
--- Copy the code
--- Go to microsoft[.]com/devicelogin
 -- Sign in with their Microsoft account
-- Browser auto-copies user code to clipboard
-Victim Authenticates    
-- Victim opens microsoft.com/devicelogin 
-- Victim pastes user code and signs in 
-Token Harvest(server-side behavior)
-- Attacker server polls Microsoft repeatedly
-- Once victim completes login, Microsoft returns access token to attacker server
-- Attacker server stores the tokens and notifies browser: session complete
- Cover Redirect    
-- Browser receives "session complete" from attacker server
-- Browser shows "Verification Complete!" then redirects victim to the legitimate brand page
-- Victim believes they completed a legitimate verification step
- Post-Compromise (server-side behavior)
-- The access token provides immediate access to the victim's Microsoft account
-- Resources at risk: email (Outlook), files (OneDrive), chat (Teams), identity (Azure AD)

The campaign includes several sophistications worth noting:
- Three versions of the attack targeting three services
- Obfuscated delivery: one variant was delivered as an AES-GCM encrypted blob via Cloudflare Workers to evade static content scanners
- Cookie exfiltration: browser cookies are also sent to the attacker on page load
- Anti-bot: the page stays completely invisible until the server confirms a human visitor; bots and crawlers are silently redirected to the real OneDrive
- Anti-analysis techniques
-- Disables right-click functionality, text selection, and drag operations.
-- Blocks keyboard shortcuts for developer tools (F12, Ctrl+Shift+I/C/J) and source viewing (Ctrl+U).
-- Detects active developer tools by utilizing a window size heuristic, which subsequently initiates an infinite debugger loop.
-- Attempts to intercept screen capture functions (PrintScreen / Win+Shift+S).
-- Obfuscates the Uniform Resource Locator (URL) with a randomly generated 132-character string following the retrieval of the email address from URL parameters.
- Scale:
-- Earliest observation on Feb 18 2026
